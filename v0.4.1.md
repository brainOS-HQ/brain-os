### Open: decision_log over-supersession on parallel batches

**Symptom.** When 5 decisions are logged in parallel (Claude batching `decision_log` calls), each falsely supersedes the previous. Reproduced live in theboards session 2026-05-21 â€” dec-019 superseded dec-018, dec-020 superseded dec-019, dec-022 superseded dec-020. Three of five decisions ended up incorrectly superseded.

**Root cause confirmed.** `decision-log.ts:46-54` auto-supersedes any active decision matching `(entity_id, type)`. Type is too coarse â€” two unrelated decisions sharing a type (e.g. both `architecture`) trigger false supersession. Sequential batch logs with same entity+type cascade.

**Scoped fix: require explicit `supersedes[]` parameter.**

1. `src/tools/decision-log.ts` (~15 lines)
   - Add `supersedes?: string[]` to `DecisionInput`
   - Replace the type-collision loop with: only supersede IDs explicitly passed in
   - Validate each ID exists and belongs to the same `entity_id`; throw on mismatch (explicit failure beats silent miss)

2. `src/server.ts` (~3 lines)
   - Add `supersedes: z.array(z.string()).optional()` to the `decision_log` schema with a clear description
   - Update the `decision_refresh` `status` description to drop the now-stale "type matching" language

**Behavior change.** Existing callers relying on auto-supersession lose it silently. This is intentional â€” the auto behavior was buggy. Callers opt in explicitly.

**Migration.** Existing decisions with stale `superseded_by` from the old buggy behavior aren't auto-repaired. The decision_refresh fix from earlier today (b672d1d) clears the pointer when reactivating. Run `decision_refresh status: "active"` on each affected decision (theboards: dec-018, dec-019, dec-020).

### Open: decision_check keyword false-positives

**Symptom.** `decision_check` can return `conflict` (hard STOP) on coincidental word matches. Two layers force STOPs on substring presence alone:
- Rejected-alternative word overlap â‰Ą50% â†’ `conflict`
- Hardcoded negation pairs (`["simple","complex"]`, `["monolith","microservice"]`, `["start","stop"]`, etc.) firing on substring presence anywhere â†’ `conflict`

The semantic layer exists but only *adds* cautions; can't downgrade the keyword conflicts.

**Hidden trap to avoid.** Current `embedDecision` concatenates chosen direction AND rejected alternatives into one embedding per decision. So `semanticRecall` can't tell "matches what was rejected" from "matches what was decided" â€” both score high.

**Scoped fix (approach C â€” directional embeddings):**

**Files to change:**

1. `src/utils/embeddings.ts` (~40 lines)
   - Add optional `facet?: "chosen" | "rejected"` to `StoredEmbedding`
   - `embedAndStore`: accept facet param; existing-entry lookup matches on (kind, id, facet)
   - `embedDecision`: write two embeddings when alternatives exist
     - `chosen` facet: `decision | why | chosen_direction | proof_action`
     - `rejected` facet: alternatives joined as `option: rejected_because`
     - Decisions with no alternatives: chosen only
   - `semanticRecall`: add optional `facet` filter

2. `src/tools/decision-check.ts` (~30 lines)
   - Demote layers 1 (rejected-alt word overlap) and 2 (negation pairs) to add to `cautions` initially, not `conflicts`
   - Replace single semantic block with two calls:
     - `rejectedMatches`: facet=`rejected`, threshold 0.65
     - `chosenMatches`: facet=`chosen`, threshold 0.5
   - Promotion logic per decision:
     - Keyword-flagged + in `rejectedMatches` â†’ promote to `conflict` (STOP)
     - Keyword-flagged + in `chosenMatches` only â†’ drop the caution (alignment, not conflict)
     - Keyword-flagged + no semantic hit â†’ stays `caution`
     - Not keyword-flagged + in `rejectedMatches` â†’ `caution` (semantic-only suspicion)
   - On `EmbeddingsNotConfiguredError`: catch, keep all keyword flags as cautions, never return `conflict`

**Migration.** Existing `embeddings.json` entries have no `facet` field. Treat unfaceted as `chosen` for back-compat. Old decisions re-embed organically next time `decision_log` or `decision_refresh` touches them.

**Manual test scenarios (no test infra in repo):**
1. Decision picks postgres, rejects redis â†’ proposed "add redis" should be `conflict`
2. Same decision â†’ proposed "tune postgres pool" should be `clear` or `caution` (chosen alignment drops keyword caution)
3. Same decision â†’ proposed "switch to mongodb" â†’ `caution` (no keyword, semantic-only)
4. Embeddings off â†’ no scenario returns `conflict`

**Size.** ~70 lines across 2 files. No schema or MCP API changes.

### Open: plan_advance over-promotion

**Symptom.** Completing a non-active step promotes the next pending step to active even when an active step already exists, leaving two steps in `active` state simultaneously. Surfaced 2026-05-19: completing step-003 wrongly promoted step-004 while step-002 was still active.

**Root cause.** `plan-update.ts:33-50 promoteNextStep` finds the first pending step and marks it active without first checking whether an active step already exists.

**Fix.** Guard `promoteNextStep` â€” if `entity.plan.find(s => s.status === "active")` returns a step, skip promotion and return null. Three lines added.
