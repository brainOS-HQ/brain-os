# v0.4.1 — Reliability Pass

Open bugs and scoped fixes for the next Brain OS release. Three bugs surfaced today (2026-05-21); one fixed, two open.

## Bug status

### Fixed today

**decision_refresh leaves stale `superseded_by`** — when status transitions away from `superseded`, the `superseded_by` pointer wasn't cleared, creating dangling references. Fixed in `src/tools/decision-refresh.ts:51-58`. Now clears the pointer on any status change away from `superseded`.

Repair existing dangling pointers in theboards project: run `decision_refresh` with `status: "active"` on dec-018, dec-019, dec-020 — new code path will clear them.

### Open: decision_log over-supersession on parallel batches

**Symptom.** When 5 decisions are logged in parallel (Claude batching `decision_log` calls), each falsely supersedes the previous. Reproduced live in theboards session 2026-05-21 — dec-019 superseded dec-018, dec-020 superseded dec-019, dec-022 superseded dec-020. Three of five decisions ended up incorrectly superseded.

**Likely cause.** Race condition in `decision-log.ts` — read decisions.json → mutate → write, with no locking. Parallel calls read the same baseline and each marks the previous as superseded based on stale state.

**Fix direction.** Either serialize writes (file lock / single-flight queue) or detect "same-batch" logging and skip auto-supersession. Not yet scoped.

### Open: decision_check keyword false-positives

**Symptom.** `decision_check` can return `conflict` (hard STOP) on coincidental word matches. Two layers force STOPs on substring presence alone:
- Rejected-alternative word overlap ≥50% → `conflict`
- Hardcoded negation pairs (`["simple","complex"]`, `["monolith","microservice"]`, `["start","stop"]`, etc.) firing on substring presence anywhere → `conflict`

The semantic layer exists but only *adds* cautions; can't downgrade the keyword conflicts.

**Hidden trap to avoid.** Current `embedDecision` concatenates chosen direction AND rejected alternatives into one embedding per decision. So `semanticRecall` can't tell "matches what was rejected" from "matches what was decided" — both score high.

**Scoped fix (approach C — directional embeddings):**

**Files to change:**

1. `src/utils/embeddings.ts` (~40 lines)
   - Add optional `facet?: "chosen" | "rejected"` to `StoredEmbedding`
   - `embedAndStore`: accept facet param; existing-entry lookup matches on (kind, id, facet)
   - `embedDecision`: write two embeddings when alternatives exist
     - `chosen` facet: `decision | why | chosen_direction | proof_action`
     - `rejected` facet: alternatives joined as `option: rejected_because`
     - Decisions with no alternatives: chosen only
   - `semanticRecall`: add optional `facet` filter

2. `src/tools/decision-check.ts` (~30 lines)
   - Demote layers 1 (rejected-alt word overlap) and 2 (negation pairs) to add to `cautions` initially, not `conflicts`
   - Replace single semantic block with two calls:
     - `rejectedMatches`: facet=`rejected`, threshold 0.65
     - `chosenMatches`: facet=`chosen`, threshold 0.5
   - Promotion logic per decision:
     - Keyword-flagged + in `rejectedMatches` → promote to `conflict` (STOP)
     - Keyword-flagged + in `chosenMatches` only → drop the caution (alignment, not conflict)
     - Keyword-flagged + no semantic hit → stays `caution`
     - Not keyword-flagged + in `rejectedMatches` → `caution` (semantic-only suspicion)
   - On `EmbeddingsNotConfiguredError`: catch, keep all keyword flags as cautions, never return `conflict`

**Migration.** Existing `embeddings.json` entries have no `facet` field. Treat unfaceted as `chosen` for back-compat. Old decisions re-embed organically next time `decision_log` or `decision_refresh` touches them.

**Manual test scenarios (no test infra in repo):**

2. Same decision → proposed "tune postgres pool" should be `clear` or `caution` (chosen alignment drops keyword caution)
3. Same decision → proposed "switch to mongodb" → `caution` (no keyword, semantic-only)
4. Embeddings off → no scenario returns `conflict`

**Size.** ~70 lines across 2 files. No schema or MCP API changes.
